3Briefing - Background Information

Project Overview

This prototype addresses a critical gap identified in the ODU RecWell bicycle rental service through ethnographic research. Currently, students must physically visit the rental desk to check bicycle availability, leading to wasted time and frustration when bikes are unavailable. Additionally, front desk staff lack an efficient tool to manage and update bicycle status as rentals and returns occur throughout the day.

The Bicycle Inventory Dashboard is a web-based solution that serves two primary user groups:

  • Students: Check real-time bicycle availability before visiting RecWell
  • Front Desk Staff: Quickly update bicycle status (available, rented, under repair)

Primary Task Scenarios

Scenario 1: Student Checking Bicycle Availability

User: Alex (Graduate Student)
Goal: Check if bicycles are available before making the trip to RecWell

Alex is a graduate student who wants to rent a bicycle for the afternoon. Before leaving their apartment, they want to check if bikes are currently available at RecWell to avoid making an unnecessary trip. Alex accesses the bicycle inventory dashboard on their phone, views the current availability, and decides whether to visit the rental desk based on real-time information.

Expected Outcome: Alex sees "5 Bicycles Available" and decides to walk to RecWell, confident that bikes are in stock.

Scenario 2: Front Desk Staff Updating Inventory Status

User: Jordan (RecWell Front Desk Staff)
Goal: Quickly update the system when bicycles are rented, returned, or need repair

Jordan works at the ODU Outdoor Adventure Center front desk. Throughout her shift, students rent bicycles and return them. Some returned bikes need repairs before being rented again. Jordan needs to quickly update the system to reflect which bikes are available, rented, or under repair so that the real-time display remains accurate for students.

Expected Outcome: Jordan clicks "Rented" on Bike #3 after a student checks it out. The public display immediately updates to show one fewer bike available.

Hierarchical Task Analysis

Task 1: Check Bicycle Availability (Student)

  1. Access the dashboard
    • Navigate to the bicycle inventory webpage
    • Wait for page to load
  2. View current status
    • Read total available bikes
    • Check last updated timestamp
  3. Make decision
    • If bikes available → plan to visit RecWell
    • If no bikes available → delay trip or make alternative plans

Plan: Do steps 1→2→3 sequentially. Stop when decision is made.

Task 2: Update Bicycle Inventory (Staff)

  1. Authenticate to staff interface
    • Enter staff credentials
    • Access inventory management panel
  2. Process bicycle transaction
    • Identify transaction type (rental, return, repair)
    • Select appropriate bicycle from list
  3. Update status
    • Click appropriate status button
    • Confirm changes
    • Verify real-time display updates

Plan: Do step 1 once at shift start, then repeat steps 2-3 for each transaction throughout the shift.

Use Cases

Use Case 1: Student Checks Availability Before Visit

Preconditions:

  • Dashboard is accessible online
  • Inventory has been updated recently by staff

Main Flow:

  1. Alex opens the bicycle dashboard URL on their smartphone
  2. System displays current availability status
  3. Alex sees "5 bicycles available" with timestamp "Last updated: 1 minute ago"
  4. Alex decides to walk to RecWell to rent a bike

Alternative Flow: No bikes available

  • Step 3a: Alex sees "0 bicycles available"
  • Step 4a: Alex decides to check back later or make other plans

Use Case 2: Staff Updates Bicycle Inventory Status

Actor: Staff Member (Jordan)

Preconditions:

  • Staff member is logged into the inventory management system
  • Bicycle records already exist in the system

Main Flow - Bicycle is Rented:

  1. Jordan opens the staff inventory management interface
  2. A student rents a bicycle from the front desk
  3. Jordan selects the bicycle in the system
  4. Jordan changes its status from "Available" to "Rented"
  5. System updates the inventory count automatically
  6. Public dashboard reflects the updated bicycle availability

Alternative Flow 2a: Bicycle is returned in good condition

  • Jordan changes status from "Rented" to "Available"
  • System increases available bicycle count

Alternative Flow 2b: Bicycle requires repair after return

  • Jordan changes status from "Rented" to "Under Repair"
  • System excludes bicycle from available inventory

Design Rationale

Student View Design

The student view prioritizes immediate information visibility. The large number (96pt font) ensures the availability count is readable at a glance, even on mobile devices. The color changes from green (bikes available) to red (no bikes) provide instant visual feedback. The breakdown section gives context without cluttering the main display. Auto-refresh every 30 seconds ensures students always see current data without manual page reloads.

Staff Management Interface Design

The staff interface optimizes for speed and efficiency. Each bike entry shows its current status with color-coded badges (green = available, blue = rented, orange = repair) and provides three buttons for quick status changes. The current status button is disabled to prevent redundant clicks. Staff can update a bike's status in under 5 seconds: click the appropriate button, see confirmation notification, done. This efficiency is critical during busy periods when multiple students are renting or returning bikes simultaneously.

Interaction Flow Examples

Example 1: Student Checks Before Visiting

  1. Student opens URL on phone: odu.edu/recwell/bikes
  2. Page loads in < 2 seconds, shows "5 Bicycles Available"
  3. Student sees "Last updated: 2:34 PM" (current time is 2:35 PM)
  4. Student decides to walk to RecWell

Total interaction time: ~5 seconds

Example 2: Staff Processes a Rental

  1. Student completes rental paperwork, assigned Bike #7
  2. Staff opens inventory panel (already logged in)
  3. Staff locates Bike #7 (currently showing "Available" badge)
  4. Staff clicks "Rented" button
  5. System shows notification: "Bike #7 status changed to Rented"
  6. Public view updates: "4 Bicycles Available" (was 5)

Total update time: < 10 seconds

Example 3: Staff Marks Bike for Repair

  1. Student returns Bike #3, reports chain issue
  2. Staff inspects bike, confirms it needs repair
  3. Staff locates Bike #3 in system (showing "Rented")
  4. Staff clicks "Repair" button instead of "Available"
  5. Bike removed from available count
  6. Students see accurate availability (bike not counted as available)

Result: System maintains accuracy - broken bikes don't mislead students

Prototype Fidelity Notes

High Fidelity - Look: The prototype uses professional colors, fonts, spacing, and modern UI design patterns that represent the final implementation.

Medium Fidelity - Feel: The prototype runs in a web browser with interactive buttons and real-time updates. Some advanced features (like actual database connectivity) are simulated.

Medium Fidelity - Breadth: The prototype supports both main scenarios (student checking availability and staff updating inventory) with all major screens and workflows.

Low Fidelity - Depth: No backend implementation. All data is stored in browser memory. Staff authentication uses hardcoded credentials. Updates are client-side only.

System Scope and Limitations

⚠️ Important Note for Evaluators

The current dashboard supports managing the status of only the bikes already added to the system. The prototype includes 12 pre-configured bicycles (Bike #01 through Bike #12) with various statuses.

This prototype does NOT support:

  • Adding new bicycles to the system
  • Removing bicycles from the system
  • Editing bicycle details (name, ID, etc.)

The focus of this prototype is on the core workflow of status updates (Available ↔ Rented ↔ Under Repair), which represents the primary daily task for front desk staff. The addition/removal of bicycles would typically be an administrative function performed less frequently by management personnel, which is outside the scope of this prototype.
